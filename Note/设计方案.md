# 房屋租赁系统 - 设计方案

## 一、项目概述

基于 Spring Boot 2.7 + MyBatis-Plus + MySQL 的房屋租赁平台，支持租客浏览租房、房东发布管理房源、管理员审核监管。核心亮点为**真假房源鉴别机制**（规则评分 + 人工审核）。

技术栈：
- 后端：Spring Boot 2.7 + MyBatis-Plus 3.5 + MySQL + WebSocket
- 工具库：Lombok、Fastjson2、EasyExcel、Commons-Lang3
- 连接池：Druid
- 邮件：spring-boot-starter-mail（忘记密码验证码）

## 二、角色与登录设计

### 2.1 角色划分

| 角色 | 说明 |
|------|------|
| 租客 | 浏览/收藏/预约看房/签约/支付租金 |
| 房东 | 发布/管理房源/收租/签约 |
| 管理员 | 审核房源、管理用户、数据统计 |

- 租客和房东不是固定角色，用户每次登录时选择身份（current_role），可自由切换
- 管理员通过 is_admin 标记，独立登录入口，不参与身份切换

### 2.2 双登录入口

| 入口 | 路径 | 说明 |
|------|------|------|
| 用户登录 | `/login` | 租客/房东登录，登录后选择身份 |
| 管理员登录 | `/admin/login` | 管理员专用，独立页面 |

- 普通用户从管理员入口登录 → 提示"非管理员账号"
- 管理员从用户入口登录 → 提示"请从管理后台登录"

### 2.3 登录流程

```
用户登录：
  输入账号密码 → 校验通过 → 检查 is_admin=0
  → 检查邮箱是否已绑定
     ├── 未绑定 → 强制跳转绑定邮箱页（发验证码确认）
     └── 已绑定 → 选择身份（租客/房东）→ 写入 current_role → 进入系统

管理员登录：
  输入账号密码 → 校验通过 → 检查 is_admin=1 → 直接进入管理后台
```

### 2.4 权限控制

- current_role=0（租客身份）：屏蔽所有房源管理接口，不能操作房源
- current_role=1（房东身份）：可发布管理房源、收租等
- is_admin=1（管理员）：访问所有管理后台接口

## 三、功能模块

### 3.1 用户模块

**已有功能：** 登录、注册

**新增功能：**

| 功能 | 说明 |
|------|------|
| 修改个人信息 | 修改昵称、头像、邮箱、手机号 |
| 修改密码 | 需验证旧密码 |
| 忘记密码 | 通过绑定邮箱发送6位验证码，5分钟过期，验证后重置密码 |
| 绑定邮箱 | 注册/首次登录时强制绑定，发验证码确认 |
| 查看个人主页 | 展示用户信息、认证状态、收到的评价 |

**忘记密码流程：**
```
用户点击「忘记密码」→ 输入注册邮箱
→ 后端生成6位验证码，存内存缓存（ConcurrentHashMap，5分钟过期）
→ 通过 spring-boot-starter-mail 发送到邮箱
→ 用户输入验证码 + 新密码 → 验证通过 → 重置密码
```

### 3.2 认证模块

分为**实名认证**和**房源认证**两种。

**实名认证流程：**
```
房东提交实名认证
  → 上传身份证正反面 + 填写真实姓名、身份证号
  → 管理员审核
     ├── 通过 → user表 is_verified=1，写入 real_name、id_card
     └── 驳回 → 通知房东原因，可重新提交
```

**房源认证流程：**
```
房东对某个房源提交认证
  → 上传房产证/租赁合同照片
  → 管理员审核
     ├── 通过 → house表 is_certified=1，真实性评分+20
     └── 驳回 → 通知原因
```

### 3.3 房源模块

| 功能 | 说明 |
|------|------|
| 发布房源 | 房东填写房源信息、上传图片，自动计算真实性评分 |
| 搜索房源 | 按区域、价格、户型、关键词等条件筛选，分页展示 |
| 房源详情 | 展示完整信息、真实性标签、访问统计、用户备注 |
| 修改/下架 | 房东管理自己的房源 |
| 访问统计 | 浏览次数、收藏数、沟通数（房东可见） |

### 3.4 真假房源鉴别机制

核心亮点，基于**规则评分系统**，简单有效，不需要机器学习。

**评分规则（满分130，归一化到0-100展示）：**

```
加分项：
+20  房东已实名认证（is_verified=1）
+20  房源已认证（is_certified=1，有房产证）
+15  图片数量 >= 3张
+15  价格在同区域均价 50%-150% 范围内
+10  描述字数 >= 50字
+10  房东历史无虚假记录
+10  联系电话格式有效
+10  地址信息完整（省市区+详细地址）
+10  图片无重复（简单hash去重）
+10  房东历史发布无虚假记录

扣分项：
-20  有经管理员认定成立的举报（未审核的举报不计入）
-30  房东曾有房源被标记虚假
-10  同一房东短时间内发布超过10条
```

**鉴别流程：**
```
1. 房东发布房源 → 系统自动算分 → verify_score
2. 分数 >= 60：自动进入「待审核」队列
3. 分数 < 60：标记「疑似虚假」，优先人工审核
4. 管理员审核通过 → 上架；驳回 → 通知房东修改
5. 租客可举报 → 举报经管理员认定后才影响评分
```

**前端展示可信度标签：**
- **已认证**（score >= 80 且审核通过）
- **待验证**（score 60-80 或待审核）
- **疑似虚假**（score < 60 或被多次举报认定）

### 3.5 举报模块

统一举报，支持举报房源、房源评论、用户互评。

**防恶意举报机制：**

1. **举报门槛**
   - 必须登录且邮箱已验证
   - 同一用户对同一目标只能举报一次
   - 每用户每天举报上限5次

2. **举报需要成本**
   - 必须选择原因分类 + 填写描述（最少20字）
   - 可选上传截图证据，有证据的举报权重更高

3. **举报不直接生效，走人工审核**
   ```
   举报 → 进入待处理队列 → 管理员审核
       ├── 举报成立 → 目标扣分/下架，举报者信誉+5
       └── 举报不成立 → 目标不受影响，举报者信誉-10
   ```

4. **举报者信誉系统**
   - 用户表增加 report_credit（默认100）
   - 信誉低于60：每周只能举报1次
   - 信誉低于30：禁止举报

### 3.6 聊天模块

基于 WebSocket 的实时聊天，支持三种会话类型。

**会话类型：**

| type | 说明 |
|------|------|
| 0 | 租客 ↔ 房东（关联房源） |
| 1 | 租客 ↔ 客服 |
| 2 | 房东 ↔ 客服 |

**消息类型：**

| msg_type | 说明 | content 存储 |
|----------|------|-------------|
| 0 | 文本 | 纯文字 |
| 1 | 图片 | 图片URL |
| 2 | 表情 | 表情编码（如 emoji:smile 或 Unicode） |
| 3 | 文件 | JSON：`{"name":"合同.pdf","url":"xxx","size":1024}` |
| 9 | 系统消息 | 提示文字 |

**WebSocket 设计：**
```
连接：ws://localhost:8081/ws/chat?userId=xxx
发消息：客户端 → 服务端：{ conversationId, content, msgType }
推送：  服务端 → 接收方：{ conversationId, senderId, content, msgType, createTime }
离线：  消息存库，标记 is_read=0，上线后拉取未读消息数
心跳：  每30秒 ping/pong 保持连接
```

**客服分配策略：**
```
用户点击「联系客服」→ 取当前在线管理员中活跃会话数最少的 → 创建会话
无人在线时 → 分配给固定兜底客服账号
```

**快捷回复：** 房东可预设常用回复模板，聊天时一键填入发送。

### 3.7 收藏与预约

| 功能 | 说明 |
|------|------|
| 收藏房源 | 收藏/取消收藏，收藏列表 |

| 预约看房 | 租客选择时间预约，房东确认/拒绝 |

### 3.8 租约管理

**签合同流程：**
```
房东创建租约 → 填写租期、月租金 → 上传合同文件
→ 租客确认签约 → status=进行中 → 自动生成每月待缴记录
```

**续租流程：**
```
租客发起续租 → 填写新截止日期 → 房东同意 → 更新 end_date → 生成新的月度待缴
```

**退租流程：**
```
租客发起退租申请 → 房东确认 → status=已退租
房东也可发起（要求租客搬离）→ 租客确认
```

租约状态流转：`待签约 → 进行中 → 已到期 / 续租中 / 退租申请中 → 已退租`

### 3.9 收租与支付

```
租客视角：
  进入「我的租约」→ 查看待缴月份 → 点击「支付」
  → 线下支付：上传转账截图作为凭证
  → 等待房东确认

房东视角：
  进入「收租管理」→ 查看每月缴费情况（待缴/已缴/逾期）
  → 收到支付通知 → 查看凭证 → 确认收款
```

支付方式：线下转账 + 上传凭证（简单实现，不接入真实支付SDK）。

### 3.10 评论与互评

**房源评论：**
- 登录用户可对房源评论（1-5星 + 文字）
- 可限制为仅预约过/租住过的用户才能评论

**用户互评：**
- 必须存在租约关系才能评价对方
- 每个租约双方只能互评一次
- 双向独立：双方都提交后才互相可见（避免报复性差评）

**举报评论：** 走通用举报流程，管理员审核后决定删除或保留。

### 3.11 拉黑功能

租户和房东可以相互拉黑。

**拉黑效果：**
- 无法给对方发消息（发送时校验）
- 无法预约对方的房源
- 无法评价对方
- 会话列表中标记为「已屏蔽」
- 可在设置中查看黑名单并解除

### 3.12 房源备注

用户可对任意房源添加私密备注，仅自己可见。每个用户对每个房源只有一条备注（可编辑覆盖）。

### 3.13 房源推荐

基于规则的推荐系统，不需要机器学习。

**推荐策略（多路召回 + 加权混合）：**

```
推荐得分 = 行为匹配分 + 热度分 + 新鲜度分

1. 行为匹配（权重最高）
   - 浏览过的房源 → 提取偏好标签（区域、价格区间、户型）
   - 收藏过的房源 → 权重更高
   - 搜索过的关键词 → 匹配房源标题/描述

2. 热度分
   - view_count × 1 + favorite_count × 3 + contact_count × 5
   - 归一化到 0-100

3. 新鲜度分
   - 7天内发布：+20
   - 30天内发布：+10
   - 更早：+0
```

**偏好自动更新：**
```
用户浏览/收藏房源时 → 异步统计最近30天行为
→ 提取最常浏览的区域（top3）、价格中位数区间、最多的户型
→ 写入 user_preference 表
```

**相似推荐：** 房源详情页底部展示同区域、相近价格、相同户型的房源。

## 七、开发计划建议

| 阶段 | 模块 | 优先级 |
|------|------|--------|
| 1 | 建表SQL + 全部实体类 + Mapper | 基础 |
| 2 | 用户模块完善（修改信息、密码、邮箱绑定、双入口登录） | 高 |
| 3 | 房源模块（CRUD + 搜索 + 访问统计） | 高 |
| 4 | 真假房源鉴别（评分算法 + 审核流程） | 高 |
| 5 | 认证模块（实名认证 + 房源认证） | 高 |
| 6 | 租约管理（签约 + 续租 + 退租） | 高 |
| 7 | 收租支付（缴费 + 确认） | 高 |
| 8 | 聊天模块（WebSocket + 消息类型 + 快捷回复） | 中 |
| 9 | 举报模块（通用举报 + 信誉分） | 中 |
| 10 | 评论/互评 | 中 |
| 11 | 收藏/预约/备注/通知 | 中 |
| 12 | 拉黑功能 | 中 |
| 13 | 推荐系统 | 低 |
| 14 | 管理员后台（用户管理 + 审核中心 + 公告 + 操作日志） | 中 |
| 15 | 管理员仪表盘 | 低 |
| 16 | 文件上传 | 贯穿各模块 |
